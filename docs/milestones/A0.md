# A0 — The correctness spine

**Goal:** train a char-level LM whose held-out loss matches a known-good baseline on the same data budget.
**The number:** held-out **bits-per-byte within 2% of nanoGPT** on the same dataset and step budget — after the gate test (overfit one batch to ~0 loss) is green.
**Time:** ~3 weeks at ~10 hrs/week.

You are proving your implementation is *correct*, quantitatively. Not fast, not clever — correct. Everything in ember stands on this.

## Week map

- **Week 1:** prime session → `data.py` → gate test written (red) → model ladder rungs (a)–(b).
- **Week 2:** rungs (c)–(e) → gate test green → `train.py` basics → first full training run.
- **Week 3:** schedule/clip/AMP/checkpointing done properly → the baseline comparison → Deepen + writeups.

## Step 1 — Data (`ember/data.py`)

- Corpus: `bin/ember data` → fetches tiny shakespeare to `data/shakespeare/input.txt`.
- Char-level vocab built from the corpus (~65 symbols). (Byte-level, vocab 256,
  is a fine alternative — the corpus is ASCII, so the metric compares either way.)
- `encode`/`decode`, and a **contiguous** 90/10 train/val split. Contiguous
  matters: shuffling tokens before splitting leaks val into train. Know why.
- `get_batch(split, batch_size, block_size, device)` → `(x, y)` with `y` shifted
  one position — the next-token targets. Getting this shift wrong is the most
  common silent bug in the whole milestone.

## Step 2 — The gate test, FIRST (`tests/test_overfit.py`)

The skeleton is in the repo with the contract in its docstring. Write it before
the model is done — it's how you'll know each ladder rung works. Tiny config
(2 layers, 2 heads, d=64, block 64), one fixed batch, a few hundred Adam steps,
assert final loss ≤ 0.05 (tighten to 0.01 once green). Add the two cheap
asserts: initial loss ≈ ln(vocab_size), and loss falling over the first 50 steps.

## Step 3 — The model ladder (`ember/model.py`)

Each rung must pass the overfit test (or its embedding-only variant) before the next:

- **(a) Embeddings → LM head only.** A bigram model in disguise.
  *Sanity check:* untrained loss ≈ `ln(vocab_size)` — ~4.17 for a 65-char vocab,
  ~5.55 for bytes. If it's not, your loss wiring is wrong *before attention even exists*.
- **(b) One causal self-attention head.** Q, K, V projections; scaled dot-product
  (`1/√d_head` — know what breaks without it); causal mask.
  *Sanity check (3 lines, keep it as a test):* attention rows sum to 1, and every
  position's weights on **future** positions are exactly 0. (Attending to *itself*
  is correct — the bug is seeing the future.)
- **(c) Multi-head** + output projection.
- **(d) The block:** pre-norm → attention → residual add; pre-norm → MLP (4×
  expansion, GELU) → residual add.
- **(e) The stack:** N blocks, learned positional embeddings, final norm, LM head
  (weight tying optional — note whether you did it, it matters for the baseline).

## Step 4 — The training loop (`ember/train.py`)

AdamW; linear warmup → cosine decay; grad clip at 1.0; val-loss estimation every
~250 iters (`model.eval()` + `torch.no_grad()`, then *back to train mode*);
checkpoint save/resume — **tested by killing a run on purpose and resuming**;
mixed precision behind a flag (fp16 + `GradScaler` on the 2080S — Turing has no
bf16; fp32 on Mac/MPS while developing).

Develop on the Mac with a tiny config; the metric run happens on the 2080S.
`bin/ember env` reports each machine's devices, bf16 support, and rough speed —
run it once per machine before hardcoding any device/dtype choice.

## Step 5 — The number

Fairness comes from matching nanoGPT's `shakespeare_char` recipe exactly —
that's the published, known-good config, so use it for **both** runs:
6 layers, 6 heads, d=384, block 256, dropout 0.2, batch 64, lr 1e-3,
warmup 100, **5000 iters** (~10.6M params).

1. **Your run:** best val loss (nats/char) → `bpb = loss / ln(2)`.
2. **Baseline:** `bin/ember baseline` — clones nanoGPT (commit pinned in
   `baselines/nanoGPT.commit`), prepares the data, runs the exact
   `shakespeare_char` config, and extracts the best val loss from the log.
   **Same machine as your run, same iters** — that's the fairness rule.
   (Expect best val loss ≈ 1.47 nats ≈ 2.12 bpb.)
3. **Pass:** `bin/ember compare --ours <num-or-log> --baseline baselines/nanogpt_shakespeare_char.log`
   — applies the 2% bar and prints both sides in nats and bits-per-byte.
   If yours is *much better*, don't celebrate — the tool will say so too:
   suspect leakage or an eval bug and investigate with the same energy you'd
   bring to being worse.

## Self-test — the forced learning, as questions

You should be able to answer these from your own build, not from the video. Any
you can't, write in `NOTES.md` as a Deepen-phase gap:

1. Why scale attention scores by `1/√d_head`? What actually goes wrong without it?
2. How does the causal mask work mechanically, and why `-inf` before softmax rather than 0 after?
3. Why residual connections? What happens to gradients in a deep stack without them?
4. Pre-norm vs post-norm — which did you build, and why is pre-norm easier to train?
5. Why does LR warmup matter, especially early with Adam-family optimizers?
6. What failure is gradient clipping guarding against, and how would you see it in a loss curve?
7. AdamW vs Adam — what's decoupled, and why does it matter?
8. fp16 vs bf16 — why does fp16 need a `GradScaler` and bf16 doesn't? Why does the 2080S force the choice?
9. Why is an untrained model's loss ≈ `ln(vocab_size)`?
10. Why must the train/val split be contiguous for language modeling?

## Silent-bug checklist (the trap, itemized)

The A0 trap is a model that trains, underperforms, and never errors. When your
number is off, check these before anything else:

- [ ] Target shift: `y` is `x` offset by exactly one
- [ ] Mask direction: positions see the past and themselves, never the future
- [ ] `1/√d_head` scaling present
- [ ] Softmax over the last dim (keys), not tokens/batch
- [ ] `optimizer.zero_grad()` every step
- [ ] `model.eval()` + `no_grad()` during val — and back to `train()` after (dropout!)
- [ ] Norm placement consistent with your pre-norm choice; final norm present
- [ ] Contiguous split; val text never seen in training batches

## When stuck

45-minute timebox → write the one-sentence question → find the *idea* in the
video (or, after your model trains, diff against nanoGPT source) → **close the
reference** → implement from your written note. Never code side-by-side with it.

If the fundamentals themselves feel shaky (backprop, why cross-entropy), the
escape hatch is Karpathy's *makemore* series — same rules: watch, close, build.

## Deepen (only after the metric is hit)

Read **Vaswani et al. 2017, "Attention Is All You Need"** → https://arxiv.org/abs/1706.03762
It lands completely differently once you've built one. Read it against your code:

- Why multiple heads instead of one wide head?
- The paper uses post-norm + warmup; you (probably) used pre-norm. Connect those.
- Sinusoidal vs learned positional embeddings — why does the choice barely matter at this scale?
